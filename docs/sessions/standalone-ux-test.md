> **이 파일을 읽고 Team Leader로서 작업을 시작하세요.**
> Wave와 독립적으로 실행. **유저가 실시간 관전**합니다.

## 1. 팀 특성

이 팀은 다른 팀과 다르게 **실시간 브라우저 테스트**를 수행합니다.
- MCP 도구 (`mcp__Claude_in_Chrome__*` 또는 `mcp__playwright__*`)로 실제 브라우저를 조작
- 유저가 화면을 직접 보면서 과정을 관전
- 코드 수정 없음 — **테스트와 피드백만** 수행
- 발견한 이슈는 문서로 정리하여 다른 팀(Phase 11, 12)에 전달

## 2. 팀 구성

| 역할 | 모델 | 실행 방식 | 담당 |
|------|------|----------|------|
| **UX Test Director** (Team Leader) | Opus (메인) | 메인 세션 | 테스트 계획, 페르소나 전환, 종합 평가, 유저 보고 |
| **Persona Navigator** | Sonnet | Task (포그라운드) | **브라우저 직접 조작**, 페르소나별 행동 시뮬레이션 |
| **Competitive Researcher** | **Opus** | Task (백그라운드) | **경쟁 사이트 실사 방문**, 기능/UX 비교 분석, 벤치마크 데이터 수집 |
| **Heuristic Evaluator** | Sonnet | Task (백그라운드) | Nielsen 10 휴리스틱 기반 UX 평가, 심각도 분류, 경쟁사 대비 격차 분석 |
| **Issue Reporter** | Sonnet | Task (백그라운드) | 스크린샷 수집, 이슈 문서화, 최종 리포트 작성 |

### 역할 상세

**UX Test Director (Leader)**
- 어떤 페르소나로, 어떤 시나리오를, 어떤 순서로 테스트할지 결정
- 각 페르소나 테스트 전에 유저에게 "지금부터 [페르소나명]으로 테스트합니다" 안내
- Navigator가 발견한 것을 실시간 해석 — "이건 왜 문제인지" 설명
- 모든 테스트 완료 후 종합 UX 평가서 작성

**Persona Navigator**
- 실제 브라우저를 조작하는 유일한 에이전트
- 페르소나의 사고방식으로 사이트를 탐색
- 각 행동 전에 **"[페르소나] 지금 ~를 하려고 합니다"** 내레이션
- 혼란스러운 부분, 막히는 부분, 기대와 다른 부분을 즉시 보고
- 스크린샷을 자주 찍어 기록

**Competitive Researcher (Opus)**
- Navigator가 CareerWiki를 테스트하는 **동시에** 경쟁 사이트를 실제 방문하여 비교
- 동일 시나리오를 경쟁 사이트에서도 수행 → 기능/UX 차이 기록
- 각 페르소나 테스트 완료 후 "CareerWiki vs 경쟁사" 비교표 제출
- 경쟁사가 잘하는 것 → CareerWiki에 도입 제안
- 경쟁사가 못하는 것 → CareerWiki의 차별화 포인트로 강조

**Heuristic Evaluator**
- Navigator와 Researcher의 결과를 종합하여 Nielsen 10대 휴리스틱으로 채점
- 각 이슈에 심각도 부여 (Critical / Major / Minor / Cosmetic)
- **경쟁사 대비 격차 분석**: 각 휴리스틱별로 "CareerWiki는 경쟁사 대비 어디 위치?"

**Issue Reporter**
- 모든 발견 이슈를 구조화된 포맷으로 문서화
- 스크린샷 + 재현 경로 + 심각도 + 개선 제안
- **경쟁사 비교 데이터** 포함하여 최종 리포트 생성

## 3. 페르소나 정의

### 페르소나 A: 고등학생 "민준" (17세)
- **배경**: 고2, 진로 고민 시작, 스마트폰 주력
- **목표**: "나한테 맞는 직업이 뭔지 알고 싶다"
- **특성**: 짧은 집중력, 어려운 단어에 바로 이탈, 시각적 요소 중시
- **테스트 포인트**: 첫 방문 → AI 추천 시작 → 결과 이해 → 직업 상세 탐색

### 페르소나 B: 대학생 "서연" (22세)
- **배경**: 대3, 전공(경영학) 살려서 취업 준비
- **목표**: "내 전공으로 갈 수 있는 직업을 구체적으로 알고 싶다"
- **특성**: 정보량 많아도 OK, 비교/분석 선호, 데스크톱 사용
- **테스트 포인트**: 전공 검색 → 전공 상세 → 관련 직업 → AI 전공 추천 → 비교

### 페르소나 C: 직장인 이직희망 "현우" (31세)
- **배경**: IT 기업 3년차, 번아웃, 전혀 다른 분야 관심
- **목표**: "지금 스킬로 갈 수 있는 다른 길이 있을까?"
- **특성**: 시간 부족, 빠른 결과 원함, 현실적 정보(연봉, 전망) 중시
- **테스트 포인트**: AI 추천 (제약조건 포함) → 결과의 현실성 → HowTo 가이드

### 페르소나 D: 학부모 "은정" (45세)
- **배경**: 중학생 자녀 진로 도움, 디지털 리터러시 보통
- **목표**: "아이가 좋아하는 걸로 뭘 할 수 있는지 알려주고 싶다"
- **특성**: 네비게이션 혼란 시 쉽게 포기, 신뢰감 중요, 큰 글씨 선호
- **테스트 포인트**: 메인 → 직업 탐색 → 카테고리 브라우징 → 자녀에게 공유할 만한 정보 찾기

### 페르소나 E: 비전공 진로전환자 "지호" (28세)
- **배경**: 문과 졸업, 개발자 전환 고민, 자격증에 관심
- **목표**: "비전공자도 개발자가 될 수 있는지, 어떤 준비가 필요한지"
- **특성**: 구체적 로드맵 원함, HowTo 콘텐츠에 관심, 검색 많이 사용
- **테스트 포인트**: 검색 "개발자" → 관련 직업들 → HowTo 가이드 → AI 추천

### 페르소나 F: 모바일 유저 "수빈" (19세)
- **배경**: 고3, 수시 전공 선택 고민, 모바일만 사용
- **목표**: "이 전공 가면 나중에 뭘 할 수 있는지"
- **특성**: 세로 스크롤 위주, 터치 인터랙션, 작은 화면
- **테스트 포인트**: 모바일 뷰포트에서 전체 사이트 탐색, 터치 타겟, 반응형

## 4. 테스트 시나리오

### 시나리오 1: 첫인상 테스트 (모든 페르소나)
```
1. careerwiki.org 접속
2. 5초간 첫인상 — 이 사이트가 뭐 하는 곳인지 이해되는가?
3. 가장 먼저 클릭하고 싶은 곳은?
4. 메뉴/네비게이션 구조 이해되는가?
→ 스크린샷 + 첫인상 코멘트 기록
```

### 시나리오 2: AI 추천 전체 플로우
```
1. AI 추천 시작 (직업 또는 전공)
2. 프로필 입력 단계 — 질문이 이해되는가? 선택지가 적절한가?
3. 인터뷰 단계 — 질문 품질, 응답 편의성
4. 대기 화면 — 로딩 중 안내가 충분한가?
5. 결과 화면 — 추천 결과 이해되는가? 납득이 가는가?
6. 상세 클릭 — 추천 이유가 설득력 있는가?
→ 각 단계별 스크린샷 + 소요시간 + 이탈 포인트 기록
```

### 시나리오 3: 검색 & 탐색
```
1. 검색바에 자유 검색어 입력 (페르소나별 다른 검색어)
2. 검색 결과 품질 — 원하는 결과가 상위에 있는가?
3. 검색 결과 → 상세 페이지 이동 — 자연스러운가?
4. 관련 직업/전공 네비게이션 — 연결이 매끄러운가?
5. 뒤로가기 → 다른 결과 클릭 — 상태 유지되는가?
→ 검색어 + 결과 순위 + 만족도 기록
```

### 시나리오 4: 콘텐츠 소비
```
1. 직업/전공 상세 페이지 진입
2. 정보 구조 — 원하는 정보를 쉽게 찾을 수 있는가?
3. 탭/섹션 전환 — 직관적인가?
4. HowTo 가이드 — 읽기 편한가? 실용적인가?
5. 출처/신뢰도 — 정보를 믿을 수 있다고 느끼는가?
→ 스크롤 깊이 + 체류 시간 + 코멘트 기록
```

### 시나리오 5: 사용자 기능
```
1. 회원가입/로그인 플로우
2. 북마크 기능
3. 댓글 작성
4. 위키 편집 (로그인 후)
5. 마이페이지 / 설정
→ 각 기능 동작 여부 + UX 코멘트 기록
```

### 시나리오 6: 엣지케이스 & 스트레스
```
1. 존재하지 않는 URL 접속 (404 처리)
2. 매우 긴 검색어 입력
3. 특수문자 포함 검색
4. 빈 검색어 제출
5. 빠른 연속 클릭 (더블클릭 등)
6. 브라우저 뒤로/앞으로 반복
7. 네트워크 느린 환경 시뮬레이션
→ 에러 처리 적절성 기록
```

### 비교 매트릭스

Competitive Researcher가 각 경쟁사에 대해 아래 항목을 채점 (1-5):

| 비교 항목 | 설명 |
|----------|------|
| 첫인상/온보딩 | 처음 방문했을 때 사이트 목적 이해도 |
| 직업 정보 깊이 | 연봉, 전망, 필요역량 등 정보의 풍부함 |
| 검색 품질 | 검색어 입력 → 원하는 결과까지 도달 속도 |
| AI/추천 기능 | 맞춤 추천의 존재 여부와 품질 |
| 모바일 경험 | 모바일에서의 사용성 |
| 시각적 디자인 | 현대적 디자인, 가독성, 브랜드 인상 |
| 콘텐츠 신뢰도 | 출처 표기, 데이터 신선도, 전문성 |
| 네비게이션 | 원하는 정보를 찾아가는 경로의 직관성 |
| 사용자 참여 | 댓글, 편집, 공유 등 인터랙션 기능 |
| 로딩 속도 | 체감 성능 |

### Researcher 작업 흐름

```
각 페르소나 테스트와 병렬로:

1. Navigator가 CareerWiki에서 "AI 추천" 테스트 중
   → Researcher는 동시에 커리어넷에서 "진로 심리검사" 수행

2. Navigator가 "직업 검색" 테스트 중
   → Researcher는 동시에 워크넷/사람인에서 동일 검색어로 검색

3. Navigator가 "직업 상세" 조회 중
   → Researcher는 동시에 O*NET에서 같은 직업 정보 비교

4. 각 비교 후 즉시 차이점 보고:
   ┌─────────────────────────────────────┐
   │ 📊 경쟁사 비교 #2                     │
   │ 항목: 직업 검색 UX                     │
   │ CareerWiki: 검색 결과 카드형, 10개 표시  │
   │ 사람인: 자동완성 + 필터 + 리스트뷰      │
   │ 격차: 자동완성/필터 부재 → 개선 필요     │
   │ 우위: 벡터 검색으로 의미 기반 매칭 가능   │
   └─────────────────────────────────────┘
```

## 6. MCP 도구 사용 가이드

### 주요 사용 도구

**브라우저 조작** (Persona Navigator가 사용):
```
mcp__Claude_in_Chrome__navigate        — URL 이동
mcp__Claude_in_Chrome__read_page       — 현재 페이지 읽기
mcp__Claude_in_Chrome__find            — 페이지 내 텍스트 검색
mcp__Claude_in_Chrome__computer        — 클릭, 스크롤, 타이핑
mcp__Claude_in_Chrome__form_input      — 폼 입력
mcp__Claude_in_Chrome__get_page_text   — 텍스트 추출
mcp__Claude_in_Chrome__upload_image    — 스크린샷 저장
mcp__Claude_in_Chrome__resize_window   — 뷰포트 변경 (모바일 테스트)
mcp__Claude_in_Chrome__read_console_messages  — 콘솔 에러 확인
mcp__Claude_in_Chrome__read_network_requests  — 네트워크 요청 확인
```

**모바일 테스트 시**:
```
mcp__Claude_in_Chrome__resize_window   — 375x812 (iPhone), 360x800 (Android)
```

**Playwright 대체** (Claude in Chrome 불가 시):
```
mcp__playwright__browser_navigate      — URL 이동
mcp__playwright__browser_snapshot      — 페이지 스냅샷
mcp__playwright__browser_click         — 클릭
mcp__playwright__browser_fill_form     — 폼 입력
mcp__playwright__browser_take_screenshot — 스크린샷
mcp__playwright__browser_resize        — 뷰포트 변경
mcp__playwright__browser_console_messages — 콘솔 확인
```

### 스크린샷 규칙
- 각 페르소나의 각 시나리오 시작/종료 시 반드시 스크린샷
- 문제 발견 시 즉시 스크린샷

## 9. 팀 시작 절차

### Team Leader가 할 일

```
1. 유저에게 테스트 계획 브리핑 (페르소나 목록, 예상 소요시간)
2. docs/ux-test/screenshots/ 디렉토리 생성
3. 페르소나 A부터 순서대로 진행
4. 각 페르소나마다:
   a. 페르소나 전환 안내 출력
   b. Persona Navigator에게 시나리오 할당 (포그라운드)
   c. Navigator가 브라우저 조작 → 유저가 실시간 관전
   d. 이슈 발견 시 즉시 이슈 카드 출력
   e. 시나리오 완료 후 Heuristic Evaluator에게 평가 요청 (백그라운드)
5. 모든 페르소나 완료 후:
   a. Issue Reporter에게 최종 리포트 작성 지시
   b. 유저에게 종합 브리핑
```

## 11. 유저 관전 가이드

이 팀은 유저가 **실시간으로 브라우저 화면을 보면서** 테스트를 관전합니다.

- **Claude in Chrome 사용 시**: 유저의 Chrome 브라우저에서 직접 조작 → 유저가 탭을 보면 됨
- **Playwright 사용 시**: 스크린샷으로 각 단계를 공유
- 유저는 언제든 **"잠깐, 그거 다시 해봐"** 또는 **"거기서 이것도 해봐"** 지시 가능
- 유저가 추가 페르소나나 시나리오를 요청하면 즉시 추가 가능
