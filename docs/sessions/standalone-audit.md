## 1. 공통 컨텍스트

### 프로젝트 개요
- **CareerWiki**: 한국어 진로 정보 위키 + AI 직업/전공 추천 플랫폼
- **Production**: https://careerwiki.org
- **Tech Stack**: Cloudflare Workers (Pages) + Hono + D1 (SQLite) + R2 + KV + Vectorize
- **AI/LLM**: Workers AI, OpenAI API (GPT-4o-mini)
- **Frontend**: TailwindCSS 3, TipTap editor
- **Build**: Vite, TypeScript strict mode
- **데이터**: 608개 전공 + 6,945개 직업

### 필수 읽기 파일
1. `CLAUDE.md` — 프로젝트 규칙 전체
2. `MASTER_PLAN.md` — 전체 Phase 목록 및 진행 상태

## 2. 팀 구성

| 역할 | 모델 | 실행 방식 | 담당 영역 |
|------|------|----------|----------|
| Team Leader | Opus (메인) | 메인 세션 | 전체 조율, 리포트 최종 작성, 유저 보고 |
| Architecture Analyst | Sonnet | Task(run_in_background) | **코드 아키텍처**: 구조, 패턴, 의존성, 기술 부채, 확장성 |
| Product Analyst | Sonnet | Task(run_in_background) | **제품 완성도**: 기능 목록 vs 실제 구현, 유저 시나리오 검증, 데이터 품질 |
| Security & Ops Analyst | Sonnet | Task(run_in_background) | **보안/운영**: 인증, 권한, 에러 핸들링, 배포, 모니터링, 성능 |

에이전트 정의: `.claude/agents/analyst.md` (공통), `researcher.md` (필요시)

### MECE 역할 분담
```
┌─ 코드 (How it's built) ─────────────────────────────────┐
│  Architecture Analyst                                     │
│  - 프로젝트 구조, 모듈 의존성, 코드 패턴                    │
│  - 기술 스택 적합성, 기술 부채, 확장성                      │
│  - 코드 품질 (타입 안전성, 테스트, 문서화)                  │
├─ 제품 (What it does) ───────────────────────────────────┤
│  Product Analyst                                          │
│  - 기능 인벤토리 (있는 것 / 없는 것 / 반쯤 된 것)          │
│  - 유저 시나리오별 완주 가능 여부                           │
│  - 데이터 완성도 (빈 필드, 품질, 커버리지)                  │
│  - 목적 대비 달성도 (진로 정보 제공 + AI 추천)              │
├─ 운영 (How it runs) ───────────────────────────────────┤
│  Security & Ops Analyst                                   │
│  - 인증/권한/입력 검증/XSS/SQL Injection                   │
│  - 에러 핸들링, 로깅, 모니터링                              │
│  - 배포 파이프라인, 캐싱, 성능                              │
│  - 비용/한도 (D1 row limits, Vectorize quota 등)           │
└──────────────────────────────────────────────────────────┘
```

### 실행 패턴
```
1. 3개 Analyst → Task(run_in_background=true) 병렬 분석
2. Team Leader: 3가지 분석 결과 수집
3. Team Leader: 통합 리포트 초안 작성
4. ★ 유저에게 리포트 제출 → 피드백 반영
5. 최종 리포트 완성
```

## 3. Phase 13 목표

사이트의 현재 상태를 **기술 컨설턴트 수준**으로 면밀히 분석하고,
**실행 가능한 개선 방향**이 포함된 종합 리포트를 작성합니다.

### 리포트는 코드 수정을 하지 않습니다
- 읽기 전용 분석 (코드, DB, 프로덕션 사이트)
- 수정/배포 없음
- 다른 Phase와 충돌 없음 → 언제든 병렬 실행 가능

## 4. 분석 범위

### Architecture Analyst 담당

#### 프로젝트 구조 분석
```
□ 디렉토리 구조 매핑 (src/, migrations/, public/, scripts/)
□ 모듈 의존성 그래프 (어떤 서비스가 어떤 서비스를 참조하는지)
□ 순환 의존성 유무
□ 파일 크기 분포 (비대한 파일 식별 — 예: index.tsx, unifiedJobDetail.ts)
□ 관심사 분리 수준 (라우트 vs 서비스 vs 템플릿)
```

#### 코드 품질
```
□ TypeScript strict mode 준수도 (any 타입 사용 빈도)
□ 타입 단언(as) 남용 여부
□ 에러 타입 vs unknown 처리
□ 데드코드 비율 (미사용 함수, 미사용 import)
□ 코드 중복도 (유사 로직 반복 패턴)
□ 네이밍 일관성 (camelCase, PascalCase 혼용)
```

#### 아키텍처 패턴
```
□ ISR 캐싱 패턴 효과성
□ 데이터 통합 (Unified Profile) 패턴 적절성
□ AI 파이프라인 (인터뷰 → 벡터검색 → 스코어링 → 리포트) 구조
□ Cloudflare Workers 제약 내 설계 적합성 (CPU 시간, 메모리, subrequest 한도)
□ 확장 포인트: 새 기능 추가 시 얼마나 쉽게 붙일 수 있는지
```

#### 기술 부채
```
□ TODO/FIXME/HACK 주석 목록 + 영향도
□ 마이그레이션 히스토리 정합성 (0001~최신)
□ 하드코딩된 값 (매직 넘버, URL, 키)
□ 테스트 코드 유무 (단위 테스트, E2E 테스트)
□ 문서화 수준 (코드 주석, API 문서, README)
```

### Product Analyst 담당

#### 기능 인벤토리
```
□ 주요 기능 전수 조사 → 각 기능의 상태 분류:
  - ✅ 완성 (정상 작동, 엣지 케이스 처리)
  - ⚠️ 부분 구현 (기본 동작하지만 미완성 부분 있음)
  - ❌ 미구현 (코드는 있지만 작동 안 함 / 스텁만 있음)
  - 🔲 계획만 (TODO/주석으로만 존재)
```

#### 유저 시나리오 검증 (프로덕션에서)
```
□ 시나리오 1: 신규 방문자 — 메인 → 검색 → 직업 상세 → 관련 정보 탐색
□ 시나리오 2: AI 추천 사용자 — AI 추천 시작 → 프로필 → 질문 응답 → 결과 확인
□ 시나리오 3: 콘텐츠 기여자 — 직업 편집 → 저장 → 리비전 확인
□ 시나리오 4: HowTo 작성자 — 드래프트 생성 → 편집 → 발행
□ 시나리오 5: 정보 탐색자 — 전공 검색 → 상세 → 관련 직업 확인
□ 각 시나리오: 완주 가능? 어디서 막히나? UX 허들은?
```

#### 데이터 완성도
```
□ 직업 데이터: 6,945개 중 필드별 채워진 비율
  - 기본 정보 (이름, 요약, 카테고리)
  - 상세 정보 (하는일, 되는법, 연봉, 전망)
  - way(진로경로) 필드 채움률
  - 소스별 데이터 비율 (CareerNet, Work24, AI, User)
□ 전공 데이터: 608개 중 필드별 채워진 비율
□ HowTo 콘텐츠: 총 개수, 평균 길이, 품질 분포
□ 벡터 임베딩: embedding_metadata 커버리지
□ 검색 인덱스: 검색 가능한 엔티티 비율
```

#### 목적 달성도
```
□ 핵심 목적 1: "한국 고등학생~대학생이 진로 정보를 쉽게 찾을 수 있다"
  → 검색 품질, 정보 접근성, 콘텐츠 양/질 평가
□ 핵심 목적 2: "AI가 맞춤 직업/전공을 추천해준다"
  → 추천 정확도, 사용자 경험, 결과 설명력 평가
□ 핵심 목적 3: "누구나 위키처럼 정보를 기여할 수 있다"
  → 편집 시스템 접근성, 기여 허들, 품질 관리 평가
```

### Security & Ops Analyst 담당

#### 보안
```
□ 인증 시스템: 세션 관리, JWT 검증, OAuth 흐름
□ 권한 체계: 일반/편집자/관리자 역할 분리
□ 입력 검증: 사용자 입력 sanitization (XSS, SQL Injection)
□ CSRF 방어: 폼 제출, API 호출
□ 파일 업로드: R2 업로드 검증 (타입, 크기, 악성파일)
□ API 레이트 리밋: 무차별 요청 방어
□ 민감 데이터: .dev.vars 관리, 환경변수 노출 여부
□ CSP (Content Security Policy) 헤더
```

#### 에러 핸들링 & 로깅
```
□ 글로벌 에러 핸들러 유무
□ API 에러 응답 일관성 (status code, message format)
□ 사용자 친화적 에러 메시지 vs 기술적 에러 노출
□ 에러 로깅/모니터링 (어디에, 어떻게)
□ 장애 시 복구 절차 (DB 백업, 롤백 방법)
```

#### 운영 & 성능
```
□ 배포 파이프라인: 빌드 → 배포 → 확인 과정
□ DB 마이그레이션 관리 체계
□ ISR 캐시 히트율 및 무효화 전략
□ Cloudflare 리소스 사용량 (D1 rows, KV reads, R2 ops)
□ 벡터 검색 성능 (Vectorize topK 한도, 지연시간)
□ 페이지 로딩 성능 (TTFB, LCP)
□ 비용 구조 및 한도 근접도
```

### 리포트 작성 원칙
```
1. 사실 기반: 코드에서 확인한 것만 기술, 추측 금지
2. 정량적: 가능한 모든 곳에 수치 제시 (비율, 개수, 시간)
3. 실행 가능: 문제만 나열하지 말고, 각 문제에 구체적 해결 방향 제시
4. 우선순위: 모든 개선 항목에 Impact(H/M/L) + Effort(H/M/L) 명시
5. 객관적: 좋은 점도 공정하게 인정, 나쁜 점도 솔직하게 기술
6. 독자 수준: 기술 의사결정자가 읽는다고 가정 (코드를 모르는 사람도 이해 가능)
```

## 9. 다른 Phase와의 관계

- Phase 13은 **읽기 전용** 분석입니다
- 코드 수정, 배포 없음
- **어떤 Phase와도 동시 실행 가능** (충돌 불가)
- 리포트 결과는 다른 Phase의 우선순위 조정에 활용 가능
