## 치명적 오류 (즉시 500 에러 또는 데이터 손실)

### [치명-1] way 필드를 배열로 저장하면 즉시 500 에러
**상황**: way 필드에 `["단계1", "단계2"]` 형식의 배열을 저장했을 때
**원인**: 프론트엔드 `formatRichText()` 함수가 way를 string으로 기대하는데 배열이 들어오면 TypeError 발생
**결과**: 해당 직업 페이지 전체가 HTTP 500으로 응답
**해결**: way는 반드시 string 타입. 단계별 내용은 하나의 문자열 안에 줄바꿈이나 번호로 구분
**방지**: JSON 작성 후 실행 전 `typeof way === 'string'` 확인. 배열로 보이면 즉시 수정

```json
// 금지
"way": ["1단계: 학교 졸업", "2단계: 시험 합격"]

// 올바름
"way": "1단계: 학교를 졸업해야 한다.[1] 2단계: 국가시험에 합격해야 한다.[2]"
```

---

### [치명-2] _sources를 문자열 배열로 저장하면 프론트에서 렌더링 불가
**상황**: `"_sources": {"way": ["커리어넷", "워크넷"]}` 형식으로 저장했을 때
**원인**: 프론트엔드 출처 렌더링 코드가 `{id, text, url}` 객체 배열을 기대
**결과**: 출처 섹션이 빈 화면으로 표시됨. HTTP 200이지만 데이터 사실상 무효
**해결**: _sources는 반드시 A등급 포맷 (`{"id":N, "text":"기관명", "url":"https://..."}`) 사용
**방지**: 저장 전 _sources 각 배열 항목에 id, text, url 3개 키가 모두 있는지 확인

```json
// 금지
"_sources": {"way": ["커리어넷", "워크넷"]}

// 올바름
"_sources": {"way": [{"id":1, "text":"커리어넷 직업백과", "url":"https://www.career.go.kr/..."}]}
```

### [치명-3] worktree에서 배포하면 main 배포 시 롤백됨
**상황**: `vigorous-meitner` 등 worktree 브랜치에서 `npm run deploy` 실행
**원인**: Cloudflare Pages는 main 브랜치 기준으로 배포. worktree 배포는 임시 상태
**결과**: 다음 main 배포 시 worktree 변경사항이 전부 덮어써짐
**해결**: 데이터 보완 작업은 항상 main 브랜치에서 수행. `git branch` 확인 후 작업 시작
**방지**: 작업 시작 시 `git branch --show-current` 실행. `main`이 아니면 체크아웃 후 진행

## 작업 효율 교훈

### [효율-1] Python 스크립트나 SQL 파일 생성 금지 — --command 직접 실행
**상황**: 10개 배치 처리를 위해 Python 스크립트 또는 `.sql` 파일을 만들어 실행
**문제**: 파일 생성·실행 왕복으로 시간 낭비. 실행 권한 오류, 경로 문제 발생 가능
**올바른 방법**: `npx wrangler d1 execute careerwiki-kr --remote --command "UPDATE ..."` 직접 실행
**예외**: SQL이 너무 길어 명령행 한계를 초과할 때만 `--file` 옵션 사용

---

### [효율-2] sidebarJobs/sidebarMajors는 반드시 DB 실존 확인 후 추가
**상황**: 직업명을 추정해서 sidebarJobs에 추가했는데 DB에 없는 직업이었음
**결과**: 사이드바 링크가 404로 연결됨
**해결**: 추가 전 반드시 `SELECT id, name FROM jobs WHERE name = '직업명';` 확인
**주의**: DB 직업명은 정확한 한국어 공식명. 약칭/별칭 주의 (예: "의사" vs "일반의사")

---

### [효율-3] JSON 작성 시 작은따옴표 이스케이프 필요 (SQL 내부)
**상황**: SQL UPDATE 명령 내부에 JSON 문자열을 삽입할 때 작은따옴표 충돌
**원인**: SQL 문자열 구분자가 작은따옴표인데 JSON 값 안에 작은따옴표가 있으면 SQL 구문 오류
**해결**: JSON 내부의 작은따옴표를 `''` (두 개)로 이스케이프하거나, JSON 텍스트에서 작은따옴표 자체를 피함

```sql
-- 오류 발생
UPDATE jobs SET user_contributed_json = '{"way": "it's a job"}' WHERE id = '...';

-- 올바름 ('' 이스케이프)
UPDATE jobs SET user_contributed_json = '{"way": "it''s a job"}' WHERE id = '...';
```

---

### [효율-4] 10개 배치 단위로 끊어서 처리, 완료 후 즉시 검증
**상황**: 50개를 한꺼번에 처리하다가 중간에 오류 발생 시 어디서 실패했는지 파악 어려움
**해결**: 10개 단위로 처리 → 각 배치 완료 후 HTTP 200 확인 → 다음 배치 진행
**이점**: 오류 발생 범위 최소화, 진행률 파악 용이, 중단 후 재시작 가능

## 데이터 보호 교훈

### [데이터-1] 기존 overviewSalary.wage 변경 금지
**상황**: 임금 정보를 "더 잘 쓴다"며 서술형 텍스트로 교체했을 때
**원인**: `overviewSalary.wage` 숫자값 + 구조화 포맷이 있어야 임금 바 차트가 렌더링됨
**결과**: 바 차트가 사라지고 서술형 텍스트만 남음. 기존보다 UX 저하
**해결**: 기존 wage + 구조화 포맷이 존재하면 절대 덮어쓰지 않는다
**방지**: 보완 전 `overviewSalary.wage`가 이미 있는지 확인. 있으면 `sal` 텍스트 보강만 허용

---

### [데이터-2] 기존 데이터 변경 시 반드시 더 나은 출처 필요
**상황**: 기존에 워크넷 공식 통계로 채워진 데이터를 웹 검색 결과로 교체
**문제**: 기존 데이터가 더 공신력 있는 출처인 경우 오히려 품질 저하
**해결**: 변경하려면 기존 출처보다 공신력 있는 출처(정부기관, 공식 통계) 필요
**방지**: UPDATE 전 기존 `_sources` 출처를 확인하고, 교체할 출처가 더 신뢰할 수 있는지 판단

## YouTube 관련 교훈

### [유튜브-1] oembed 검증 없이 추가하면 삭제된 영상 링크가 쌓임
**상황**: YouTube URL을 검색 결과에서 바로 복사해 추가
**결과**: 비공개 전환되거나 삭제된 영상이 youtubeLinks에 포함됨. 사용자가 재생 불가 영상 클릭
**해결**: 추가 전 반드시 oembed 확인

```
WebFetch: https://www.youtube.com/oembed?url=https://www.youtube.com/watch?v=VIDEO_ID&format=json
```

200 응답 + title 반환 = 정상 영상. 오류 응답 = 제외

---

### [유튜브-3] youtubeLinks title 없으면 URL이 제목으로 노출됨
**상황**: `{"url": "https://..."}` 형식으로 title 없이 저장
**결과**: UI에서 YouTube URL 전체 문자열이 제목 자리에 표시됨
**해결**: `{"url": "...", "title": "영상 제목"}` 형식 필수. oembed 응답 `title` 값을 사용
**방지**: youtubeLinks 작성 시 title 필드 포함 여부 항상 확인

---

### [유튜브-4] 진로 가이드/인터뷰/전망 영상 우선
**상황**: 조회수 높은 유명 유튜버의 일과 vlog를 선택
**문제**: 진로 탐색 중인 사용자에게 vlog는 참고 가치가 낮음
**해결**: "이 직업이 뭔지", "어떻게 되는지", "전망은 어떤지" 관점의 영상 우선
- 우선순위: 진로 가이드 > 현직자 인터뷰 > 직업 전망 > 자격증 준비
- 비추천: 단순 일과 vlog, 개인 일상, 주제와 관련 없는 콘텐츠

---

### [유튜브-2] 영상 제목은 oembed 응답값 사용, 기억에 의존 금지
**상황**: 유튜브 영상 제목을 직접 입력했는데 실제 제목과 달랐음
**원인**: 영상 제목은 업로더가 수정 가능. 기억·추정 제목은 오래된 정보일 수 있음
**해결**: oembed 응답의 `"title"` 필드 값을 그대로 사용

```json
// oembed 응답 예시
{"title": "간호사가 직접 말하는 현실 직업 이야기", "author_name": "직업유튜버", ...}

// youtubeLinks에 이 title 사용
{"url": "https://www.youtube.com/watch?v=...", "title": "간호사가 직접 말하는 현실 직업 이야기"}
```

## 필드 구분 교훈

### [필드-1] way와 detailReady 렌더링 위치 혼동
**상황**: detailReady에 서술형 진로 경로를 작성하거나, way에 교육과정 목록을 나열

**결과**: 잘못된 섹션에 내용이 표시됨
**해결**:
- `way` → 페이지에서 **"되는 방법" 섹션**으로 렌더링 (서술형 진로 경로. 자격요건·시험·진입경로 서술)
- `detailReady` → **교육과정/채용/훈련 섹션**으로 렌더링 (curriculum/recruit/training 구조화 배열)
**방지**: way에는 서술형 문장, detailReady에는 배열 항목 목록

### [출처-1] 도메인 수준 URL은 출처로 인정되지 않음
**상황**: `"url": "https://www.career.go.kr"` 처럼 도메인만 입력
**문제**: 사용자가 클릭해도 메인 페이지로만 이동. 실제 정보를 찾을 수 없음
**해결**: 직업별 상세 페이지 URL 사용. 검색이 필요하면 검색 결과 URL이라도 포함
**기준**: URL을 클릭했을 때 해당 직업 정보가 직접 표시되어야 함

---

### [출처-3] _sources id 순서가 본문 [N] 등장 순서와 불일치
**상황**: 캐릭터디자이너 보완 시 _sources의 id 번호 부여 순서가 본문에서 [N]이 처음 등장하는 순서와 달랐음
**원인**: 각 섹션(_sources 키)을 따로 작성하다 보니 본문 등장 순서가 아닌 섹션 구조에 맞춰 id를 매겼음
**결과**: 프론트엔드에서 참조번호 [N] 클릭 시 엉뚱한 출처가 표시됨 (id가 본문 위치와 불일치)
**해결**: 본문 텍스트를 위에서 아래로 스캔하며 새 [N]이 처음 등장하는 순서대로 id를 1부터 부여
**방지**: 검증 시 본문 전체를 위→아래 읽으며 새로운 [N]이 나올 때마다 메모 → 그 목록 순서 == sources id 순서인지 대조

```
// 검증 예시
본문 스캔: [1] at way → [2] at way → [3] at sal → [4] at sal → [5] at prospect ...
_sources id 순서도: way=[1,2], overviewSalary.sal=[3,4], overviewProspect.main=[5...] ✅
```

---

### [출처-2] [N] 참조번호와 _sources id는 반드시 1:1 매핑
**상황**: 본문에 `[3]`이 있는데 _sources에 `"id": 3`인 항목이 없었음
**결과**: 프론트엔드에서 참조번호 클릭 시 출처가 표시되지 않음
**해결**: 본문 작성 후 _sources 작성 전, 사용된 모든 [N] 번호 목록을 만들고 빠짐없이 id 부여
**검증**:
1. 본문에서 `[숫자]` 패턴 모두 추출
2. _sources에서 해당 id가 모두 존재하는지 확인
3. 반대로 _sources에 있는 id가 본문에서 참조되는지 확인 (고아 id 금지)
